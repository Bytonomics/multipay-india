# MultiPay Adapter — Roadmap

Tasks and planned features for the multipay-india/multipay-go library.

## Planned Tasks

| # | Title | Description | Status |
|---|-------|-------------|--------|
| 1 | **Background polling reconciliation job** | Implement a background polling mechanism for payment status reconciliation. The primary payment verification path is webhooks, but webhooks can be delayed or lost (server downtime, network issues). A background Go routine with `time.Ticker` should periodically check orders that are still in "created" status after a configurable timeout (e.g., 30 minutes). The poller should: (a) query pending orders from the caller's store, (b) call `client.Payments().GetPayment()` or `client.Orders().GetOrder()` for each, (c) if status changed (e.g., "paid"), trigger the same business logic as the webhook handler, (d) stop polling after a configurable max timeout (e.g., 24 hours). Design considerations: the poller should be opt-in (not started by default), configurable intervals, and safe for multiple instances (distributed lock or leader election). This is a resilience feature, not a primary flow. | Planned |
| 2 | **Cashfree Subscription API integration** | Cashfree has a full Subscription API (separate from their PG SDK) at `api.cashfree.com/api/v2/subscriptions/`. It supports: subscription creation, authorization, pause, resume, cancel, charge; payment methods include UPI Autopay, eNACH/eMandate, card Standing Instructions; lifecycle states: INITIALIZED → ACTIVE → ON_HOLD → PAUSED → COMPLETED → CANCELLED. The current vendored `cashfree_pg` SDK (v0.3.0) does NOT include subscription endpoints — it only covers the Payment Gateway. To integrate: either (a) vendor Cashfree's subscription SDK if one exists in Go, or (b) implement direct HTTP calls to the subscription API endpoints using the same auth headers (`X-Client-Id`, `X-Client-Secret`). This would add a `SubscriptionProvider` interface to `ports/providers.go`, a `SubscriptionService` to `orchestration/`, capability entries (`CapSubscriptionCreate/Fetch/Cancel/Pause/Resume/Charge`) for Cashfree in the matrix, and Cashfree adapter methods. Razorpay already has subscription support in its SDK (`resources.Subscription`). Once both providers support subscriptions, it becomes a core shared surface. Reference: [Cashfree Subscription API](https://documenter.getpostman.com/view/21543638/2s93sf3rJB), [Cashfree Recurring Payments](https://www.cashfree.com/recurring-payment/), [Cashfree Subscription Payment Methods](https://www.cashfree.com/docs/payments/subscription/payment-modes). | Planned |
| 3 | **Plan sync API — idempotent multi-vendor plan management** | Both Razorpay and Cashfree require plans as a prerequisite to subscriptions. Razorpay: Plan is a HARD requirement — `Subscription.Create` requires `plan_id`. Plans are created via `client.Plan.Create(data)` which returns a `plan_id`. Plans are IMMUTABLE — no Update API exists. To change amount/interval, create a new plan. Cashfree: Plans can be passed INLINE during subscription creation (no separate step), OR pre-created via the Plan API and referenced by `planId`. Pre-created plans are recommended for reuse. **Proposed API:** A `SyncPlans()` method that takes a list of our canonical plans and ensures they exist on each configured vendor. Idempotency strategy: the caller maintains a mapping `our_plan_id → {razorpay_plan_id, cashfree_plan_id}`. If mapping exists, skip create. If missing, create on vendor and return the vendor plan ID for the caller to store. Update is NOT a universal capability (Razorpay doesn't support it) — must be capability-gated. **Implementation needs:** (a) `domain.Plan` and `domain.CreatePlanRequest` types, (b) `PlanProvider` interface in `ports/providers.go` with Create/Fetch/List, (c) `PlanService` in `orchestration/`, (d) Razorpay adapter methods wrapping `client.Plan.Create/Fetch/All`, (e) Cashfree adapter methods wrapping the Subscription API's plan endpoints. The capability matrix already has `CapPlanCreate/Fetch/List` entries but they are ghost capabilities — declared but not wired to any code. References: [Razorpay Create Plan](https://razorpay.com/docs/api/payments/subscriptions/create-plan/), [Razorpay Go SDK Plan docs](https://github.com/razorpay/razorpay-go/blob/master/documents/plan.md), [Cashfree Subscription API](https://documenter.getpostman.com/view/21543638/2s93sf3rJB). | Planned |
| 4 | **UPI Autopay for recurring subscriptions (India)** | V1 supports card-based recurring only. V2 should add UPI Autopay mandate authorization for Indian customers. Implementation needs: (a) Add UPI as a payment method option in `CreateSubscriptionRequest` — Cashfree already supports UPI via `SubsCreatePayment` with `payment_method: "upi"`, Razorpay supports UPI mandates via their subscription flow. (b) Handle UPI-specific webhook events: `CUSTOMER_PAUSED` (customer paused mandate from UPI app — only the customer can resume), `CUSTOMER_CANCELLED` (customer cancelled from UPI app — irreversible). (c) Handle `BANK_APPROVAL_PENDING` status (Cashfree-specific — auth done but bank hasn't confirmed yet). (d) The Studio Subscriptions service eligibility check needs updating: `if billingCountry == "IN" → allow card + UPI; else → cards only`. (e) The multipay-ts picker UI needs a payment method selection step when UPI is available. (f) eNACH/NACH (bank account debit) is deferred beyond V2 due to additional compliance requirements. Reference: doc 145 Contract 4 `SubscriptionStatus` enum already includes `CUSTOMER_PAUSED`, `CUSTOMER_CANCELLED`, `BANK_APPROVAL_PENDING` states in anticipation. See doc 144 section 10.13 for eligibility architecture. | Planned |
| 5 | **Full admin billing suite** | V1 admin has read views + basic actions (cancel, recovery link, webhook replay). V2 should add: (a) Bulk operations — batch cancel subscriptions, batch migrate plans. (b) Dashboard metrics — MRR, churn rate, payment success rate, retry rate, active subscription count gauge. (c) Webhook replay queue with batch processing and retry policies. (d) Billing support runbook integration. (e) Searchable correlation across org_id, subscription_id, provider_payment_id, webhook event_id. Reference: doc 143 Stage 6 has the full admin/support/ops scope. See doc 144 section 10.14 for V1 admin endpoints. | Planned |
| 6 | **Order creation should return payment URL where possible** | Currently, Cashfree's `CreateOrder` returns a `SessionID` (for frontend JS SDK checkout) but not a direct payment URL. Razorpay's `CreateOrder` also returns an order object without a direct URL — Razorpay expects the frontend to use their Checkout.js with the order ID. However, both providers support Payment Links which DO return a direct URL (`LinkURL`). For a backend-controlled flow (like OAuth-style redirect, email payment reminders, subscription renewal emails), the library should offer a convenience method or document the recommended pattern: create a Payment Link (not an Order) when you need a shareable URL. Alternatively, investigate if Cashfree/Razorpay have any way to generate a hosted checkout URL from an Order ID (Cashfree's `payment_session_id` can be used to construct a URL: `https://cashfree.com/pg/checkout/{session_id}`). The goal: backend should be able to get a clickable payment URL for ANY flow without exposing session IDs to the frontend. This keeps payment flow server-controlled, like OAuth redirects. | Planned |
